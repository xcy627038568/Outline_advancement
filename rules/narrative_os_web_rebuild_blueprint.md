# Narrative OS Web 重构蓝图（V4 章节作战台版）

## 1. 重构结论

当前 `narrative_os` 的主要问题不是“页面不够精致”，而是产品定位仍停留在“数据库展示后台”阶段，没有对齐 V4 细纲驱动写章闭环。

重构后的 Narrative OS 必须从“全局展板中心”切换为“当前章中心”：

- 首页不是总览，而是 `当前章作战台`
- 一级导航围绕“写一章”组织，不再围绕“展示一个库”组织
- 后端不再只有查表接口，而要提供工作流聚合接口
- 前端不再只显示事实结果，而要显示“写前上下文 + 写中动作 + 写后闭环状态”

## 2. 新产品目标

### 2.1 核心目标

Narrative OS Web 的核心任务只有一个：

`帮助操作者围绕当前章节，完成 情报收集 -> 正文写作 -> 台账整理 -> 闭环回写 的完整动作链。`

### 2.2 非目标

以下能力不是第一阶段重点，不应反客为主：

- 花哨图谱首页
- 纯 BI 型总览页
- 与当前章无关的炫示统计
- 大而全的手工录入后台

## 3. 新信息架构

### 3.1 一级导航

- `/`：当前章作战台
- `/chapters`：章节库
- `/outline`：细纲源文件
- `/entities`：角色与实体
- `/assets`：资产与资源
- `/reports`：审计与汇总

### 3.2 当前章作战台布局

采用三栏布局：

- 左栏：章节列表、当前状态、切章入口
- 中栏：本章核心战术信息
  - 本章概况
  - 雷达视野
  - 上一章承接
  - 本章角色上下文
  - 本章资产快照
- 右栏：执行与产物
  - 正文预览
  - 台账预览
  - 工作流动作
  - 执行日志/状态

### 3.3 写作者视角命名原则

当前章页不能继续直接暴露数据库字段名或脚本内部概念，必须统一改成写作者能立即理解的业务命名：

- `silver_taels` 这类资产字段必须显示为 `银子`
- `treasure_notes` 这类资产字段必须显示为 `银票`
- `written_summary`、`next_hook`、`key_assets_change` 等字段只允许以中文业务名出现在页面上
- 页面区块必须围绕任务命名，而不是围绕技术来源命名

建议当前章页区块统一命名为：

- `本章任务卡`
- `本章正文`
- `本章台账`
- `上一章承接`
- `正文实际出场角色`
- `角色补充申请`
- `资产快照`
- `执行日志`

## 4. 后端重构目标

### 4.1 设计原则

- `章节优先`：先返回当前章所需聚合数据，再谈全局表
- `动作聚合`：把多个脚本和数据库读取结果整理为前端可直接消费的 JSON
- `只读先打通`：第一阶段先打通读链路，第二阶段再补写链路和闭环动作
- `主库唯一`：以项目根目录 `novel_ledger.db` 为事实源

### 4.2 第一阶段接口

- `GET /api/workflow/current`
  - 返回“当前应处理章节”的聚合上下文
- `GET /api/workflow/{chapter_no}`
  - 返回指定章节的聚合上下文
- `GET /api/chapters`
  - 保留，用于左栏导航
- `GET /api/chapters/{chapter_no}`
  - 保留，用于章节详情和正文回看

### 4.3 第二阶段接口

- `POST /api/workflow/{chapter_no}/radar`
- `POST /api/workflow/{chapter_no}/recent-context`
- `POST /api/workflow/{chapter_no}/character-context`
- `POST /api/workflow/{chapter_no}/asset-context`
- `POST /api/workflow/{chapter_no}/save-draft`
- `POST /api/workflow/{chapter_no}/backup`
- `POST /api/workflow/{chapter_no}/finalize`
- `GET /api/workflow/{chapter_no}/state`

### 4.4 第二阶段补充约束

- 角色上下文的主来源改为 `正文实际命中角色`
- 只有在正文为空或未命中角色时，才允许回退到细纲/雷达兜底
- 需要新增 `角色补充申请` 接口，允许页面按角色名主动补拉角色资料
- 资产接口返回值不能只给原始字段，必须同时返回 `中文显示名 + 单位 + 分组中文名`

## 5. 前端重构目标

### 5.1 第一阶段

- 首页切换为“当前章作战台”
- 引入统一 API 层，不再写死 `http://localhost:8000/api`
- 当前章页以聚合接口为主，不再前端拼装零散请求
- 保留旧页面作为辅助浏览入口，但从主流程降级

### 5.2 第二阶段

- 加入正文编辑区与草稿保存
- 加入台账编辑器
- 加入闭环执行状态条
- 加入失败日志和错误提示

### 5.3 交互重构要求

- 正文与台账都改为 `查看态 / 编辑态`
- 默认只显示查看态，点击后才进入编辑态
- `保存`、`放弃修改`、`编辑` 按钮必须放在对应 section 内部
- 一旦内容被修改但未保存，页面必须显示 `未保存` 状态
- 切章、刷新、闭环前都必须检查未保存状态
- 闭环前如果正文或台账未保存，必须阻止继续执行并提醒手动保存

## 6. 数据分层

### 6.1 事实层

来自主库，用于长期记录：

- `chapters_nav`
- `entities_registry`
- `entity_relationships`
- `wealth_and_assets`
- `hooks_network`
- `world_facts`

### 6.2 上下文层

来自脚本和临时预览文件，用于当前章动作：

- 雷达文件
- 上一章承接信息
- 角色上下文
- 资产快照
- 正文草稿
- 台账草稿
- 执行日志

## 7. 分阶段实施顺序

### 阶段 A：骨架纠偏

- 新建本蓝图文档
- 首页改为当前章作战台
- 抽离统一 API 客户端
- 增加工作流聚合接口

### 阶段 B：写前上下文打通

- 当前章页接入雷达
- 接入上一章摘要/钩子
- 接入角色上下文
- 接入资产快照

### 阶段 C：写中动作打通

- 草稿保存
- 原稿备份
- 台账编辑
- 台账校验

### 阶段 D：闭环执行打通

- 一键 finalize
- 状态刷新
- 结果回显
- 异常日志可视化

### 阶段 E：写作者视角校正

- 资产字段中文化与单位化
- 正文/台账双态编辑
- 未保存提醒
- 角色来源改为正文优先
- 角色补充申请入口

## 8. 本次执行范围

本轮立即执行以下改造：

1. 在 `rules` 下落盘本蓝图
2. 后端增加第一阶段工作流聚合接口
3. 前端增加“当前章作战台”页面并设为首页
4. 前端引入统一 API 基础层，消除写死地址
5. 保留旧页面，避免一次性推翻所有浏览功能

## 9. 验收标准

- 打开首页即可看到当前章作战台，而不是旧总览
- 能在一个页面看到章节列表、章节详情、雷达、前情、角色、资产、正文
- API 地址支持通过环境变量配置
- 前后端能正常构建运行
- 旧页面不因本次调整而整体失效
- 页面不再直接裸露 `silver_taels` 这类数据库字段
- 正文与台账不会同时重复显示“预览 + 编辑”两份完整内容
- 角色区默认显示正文实际出场角色，并支持页面内补申请角色资料
